# ROS 2 Jazzy 与 MuJoCo 学习项目说明

> 本文用于在新的 ChatGPT 对话中快速恢复项目上下文。后续讨论应以本文为基础，并根据实际进度持续更新。

## 1. 我的学习目标

我是只能在电脑上进行机器人仿真的学生，希望通过一个完整项目学习 ROS 2 Jazzy 和 MuJoCo，而不是只零散运行教程。

核心目标是掌握：

- ROS 2 的节点、Topic、Service、Action、Parameter；
- ROS 2 工作空间、Package、Launch、YAML 和自定义接口；
- URDF/Xacro、TF2、RViz2 和 rosbag2；
- MuJoCo 的 MJCF 模型、关节、执行器、传感器、接触和仿真循环；
- ROS 2 与 MuJoCo 的通信和控制链路；
- ros2_control 的标准控制框架；
- MoveIt 2 的机械臂运动规划；
- 相机感知、坐标转换和自动抓取任务；
- 一个完整机器人软件工程的组织、调试和运行方法。

我不以深入研究 ROS 2 或 MuJoCo 源码为第一目标，而是希望先做通一套完整系统，再逐步理解和替换其中的模块。

## 3. 主学习项目

主项目为：

> **基于 ROS 2 Jazzy 与 MuJoCo 的机械臂视觉抓取和分拣系统。**

计划在 MuJoCo 中建立一台 4～6 自由度机械臂、两指夹爪、桌面、若干彩色方块和分拣区域。系统最终应能识别桌面物体，并将不同物体抓取到指定位置。

完整任务流程：

1. MuJoCo 模拟机械臂、夹爪、物体、相机和接触；
2. ROS 2 接收关节状态和相机数据；
3. 感知节点确定目标物体的位置；
4. TF2 将物体位置转换到机械臂基座坐标系；
5. MoveIt 2 规划机械臂运动；
6. ros2_control 将关节轨迹发送到 MuJoCo；
7. 夹爪抓取物体并移动到对应分拣区域；
8. ROS 2 Action 汇报任务阶段、进度和执行结果；
9. rosbag2 记录图像、关节状态、目标位置和控制指令。

系统的大致数据流为：

```text
MuJoCo 仿真
    ↓ 关节状态、相机图像
ROS 2 感知与状态节点
    ↓ 目标位姿
TF2 坐标转换
    ↓
抓取任务状态机
    ↓
MoveIt 2 运动规划
    ↓ 关节轨迹
ros2_control
    ↓ 控制量
MuJoCo 执行器
```

## 4. 建议的软件包结构

```text
mujoco_arm_ws/src/
├── arm_description/          # URDF、Xacro、MJCF、模型和 RViz 配置
├── arm_mujoco/               # MuJoCo 场景、仿真节点和启动文件
├── arm_interfaces/           # 自定义 msg、srv 和 action
├── arm_control/              # ros2_control 与控制器配置
├── arm_perception/           # 相机、OpenCV 和物体定位
├── arm_motion_planning/      # MoveIt 2 配置与规划节点
├── arm_tasks/                # 抓取、放置和分拣状态机
└── arm_bringup/              # 整套系统的一键启动入口
```

不同模块应尽量通过标准 ROS 2 接口连接，避免把仿真、识别、规划和任务逻辑全部写在一个程序中。

## 5. 分阶段学习路线

### 阶段一：MuJoCo基础

- 使用 MJCF 建立简单二关节机械臂；
- 学习 `body`、`joint`、`geom`、`actuator` 和 `sensor`；
- 理解 `qpos`、`qvel`、`ctrl` 和 `mj_step()`；
- 使用 Python 读取状态并控制关节；
- 尝试位置控制和简单 PD 控制；
- 观察质量、惯量、阻尼、摩擦和仿真步长对系统的影响。

阶段成果：不依赖 ROS 2，机械臂能够在 MuJoCo 中按目标关节角运动。

### 阶段二：ROS 2基础通信

- 学习 Workspace、Package 和 `colcon build`；
- 编写 Publisher、Subscriber 和 Timer；
- 学习 Topic、Service、Action 和 Parameter；
- 编写 Launch 和 YAML 配置；
- 制作一个不依赖 MuJoCo 的虚拟关节控制示例。

阶段成果：目标角节点发布指令，虚拟机械臂节点发布关节状态，监控节点显示控制误差。

### 阶段三：手写最小MuJoCo—ROS 2接口

- 编写 `mujoco_sim_node`；
- 订阅目标关节指令；
- 将指令写入 MuJoCo 的 `data.ctrl`；
- 执行仿真步进；
- 发布 `/joint_states`、传感器数据和 `/clock`；
- 理解物理仿真频率与 ROS 2 控制频率的区别。

建议先手写一次最小接口，以真正理解两个系统如何连接，之后再切换到标准 ros2_control 接口。

### 阶段四：URDF、TF2与RViz2

- 为同一台机械臂建立 URDF/Xacro；
- 保证 URDF 与 MJCF 的关节名称、零位、轴方向和限制一致；
- 使用 `robot_state_publisher` 发布 TF；
- 在 RViz2 中显示机器人、关节和坐标系；
- 确保 MuJoCo 和 RViz2 中的姿态同步。

### 阶段五：ros2_control

- 学习 Hardware Interface、State Interface 和 Command Interface；
- 使用 Controller Manager；
- 使用 Joint State Broadcaster；
- 使用 Joint Trajectory Controller；
- 将 MuJoCo 包装成仿真硬件接口；
- 通过标准 FollowJointTrajectory Action 控制机械臂。

### 阶段六：MoveIt 2运动规划

- 配置机械臂 Planning Group 和末端执行器；
- 学习 SRDF、逆运动学和碰撞模型；
- 在 RViz2 中设定末端目标；
- 规划关节空间和笛卡尔路径；
- 将轨迹交给 ros2_control，并由 MuJoCo 执行。

这一阶段只要求机械臂可靠到达指定位置，暂时不加入视觉和抓取。

### 阶段七：夹爪与已知位置抓取

- 在 MuJoCo 中加入两指夹爪、桌面和方块；
- 学习接触、摩擦和夹爪执行器；
- 第一版直接由 MuJoCo 提供物体真实位置；
- 编写预抓取、下降、闭合、抬升、移动和释放流程；
- 完成一次已知坐标的抓取与放置。

### 阶段八：视觉感知与TF2

- 添加 RGB 或 RGB-D 相机；
- 将相机画面发布为 ROS 2 Image；
- 第一版使用 OpenCV 颜色分割识别彩色方块；
- 计算物体在相机坐标系中的位置；
- 使用 TF2 转换到 `base_link`；
- 发布 `PoseStamped` 目标位姿。

后续可以将颜色识别替换为目标检测模型，但第一版不需要使用神经网络。

### 阶段九：自动分拣任务

- 编写抓取和分拣 Action Server；
- 使用状态机组织抓取流程；
- 持续反馈当前阶段和完成进度；
- 支持任务取消、执行失败和有限次数重试；
- 使用 Service 实现场景重置和重新生成物体；
- 使用参数配置速度、抓取高度、安全距离和重试次数。

### 阶段十：工程化整理

- 使用一个 Launch 文件启动整套系统；
- 使用 YAML 管理参数；
- 使用 rosbag2 记录关键数据；
- 添加日志和异常处理；
- 整理 Git 仓库和 README；
- 随机改变物体位置、质量和摩擦系数，统计抓取成功率。

## 6. 第一版项目边界

为了避免项目初期过于复杂，第一版明确采用以下简化：

- 先做 2 自由度模型学习 MuJoCo，再升级到 4～6 自由度机械臂；
- 先使用位置控制，再学习力矩控制；
- 先手写简单 ROS 2—MuJoCo 接口，再使用 ros2_control；
- 先使用物体真实坐标完成抓取，再加入相机识别；
- 先使用颜色分割，不立即使用 YOLO；
- 先完成固定桌面的单次抓取，再做随机位置和连续分拣；
- 暂时不考虑真实机器人和 Sim2Real；
- 暂时不加入强化学习、SLAM、Nav2、多机器人和复杂具身智能任务。

## 12. 每次任务结束时的更新规则

后续 ChatGPT 在完成一次任务后，应根据实际情况更新本文，至少检查以下内容：

1. 更新“项目进度总表”中的复选框；
2. 更新“当前阶段、当前任务、最近完成和下一步”；
3. 更新环境版本、工作空间路径等实际信息；
4. 把新建或修改的重要文件加入“当前已产出的文件”；
5. 把新出现的问题加入“未解决问题”；
6. 问题解决后移入“已解决问题”；
7. 把影响后续实现的选择加入“关键决定”；
8. 在“对话更新日志”中增加一条简洁记录；
9. 不得把仅讨论、未运行或未验证的内容标记为完成；
10. 更新后应将同一份 Markdown 文件交还给我，供下一次对话继续使用。

## 13. 给后续ChatGPT的协作要求

在新的对话中，请按照以下方式协助我：

1. 先询问或确认我当前做到哪个阶段，不要默认所有环境和依赖已经配置完成；
2. 每次只推进一个明确的小目标，并给出完成标准；
3. 命令要说明在哪个终端、哪个目录、哪个环境中执行；
4. 对新概念先解释它在整个系统中的位置，再给代码；
5. 提供代码时应给出文件路径，并说明它属于哪个 ROS 2 Package；
6. 出错时优先检查环境、Topic、TF、时间戳、QoS、参数和节点状态；
7. 不要一开始引入强化学习、复杂视觉模型或真实硬件；
8. 每完成一个可验证步骤，按照第 12 节的规则更新本文，而不是只在完成整个阶段后更新；
9. 新对话开始时，以第 9 节记录的当前状态为准，但仍应询问我是否在其他地方继续做过；
10. 如果本文记录与我提供的实际终端输出冲突，以实际输出为准，并同步修正文档。

## 14. 新对话使用方法

新对话开始后，可以直接说：

> 请阅读这份项目说明，先根据“当前工作状态”和“项目进度总表”判断我做到哪里。询问我是否有尚未记录的新进展，然后从“下一步”继续。每次只推进一个明确任务，任务完成后更新同一份 Markdown 进度文档。
