# pj1_ws — Codex 协作规则

## 项目范围

- 本目录 `pj1_ws/` 是 ROS 2 入门学习工作空间；后续 ROS 2 包、配置、启动文件和文档均应放在这里。
- 当前工作空间仅包含本文件；尚无 `src/`、ROS 2 Package、构建配置、Launch 文件或测试。不要将这些目录、接口或命令表述为已存在。
- 当前目标是 ROS 2 基础学习：先完成不依赖 MuJoCo 的虚拟关节控制示例，再进入 MuJoCo—ROS 2 接口。优先完成 Workspace、Package、Publisher、Subscriber、Timer、Parameter、Service、Action、Launch 和 YAML。
- ROS 2 Jazzy 是计划使用的版本，但尚未在本工作空间验证；操作系统、ROS 2、MuJoCo、colcon、rosdep 和 IDE 的安装状态也均未确认。
- 在基础阶段暂不引入 MuJoCo、ros2_control、MoveIt 2、复杂视觉、强化学习、SLAM、Nav2 或真实硬件，除非用户明确要求进入对应阶段。
- 不要修改工作空间外的学习资料，除非用户明确要求。

## 项目记录

- 开始新任务时，先以项目说明中的“当前工作状态”和“项目进度总表”为依据，并询问用户是否有未记录的新进展。

## 工作方式

- 每次任务先阅读相关文件、确认现状和影响范围；不假定 ROS 2、依赖、包或工作空间已经存在。
- 先给出最小实施计划；一次只执行一个可验证步骤。用户要求只规划、检查或解释时，绝不修改文件。
- 保持改动最小；不要进行无关重构、批量格式化或改名；保留已有接口、默认行为和用户的未提交改动。
- 新增或修改功能后，说明文件路径、用途、实际执行的验证命令及结果。

- 对新概念先说明其在整体系统中的作用；提供代码时说明文件路径和所属 ROS 2 Package。

## ROS 2 约定

- 优先采用标准 ROS 2 接口与命名：`sensor_msgs/msg/JointState`、Topic、Service、Action、Parameter、Launch 和 YAML。
- 节点职责应单一；通信接口、参数和配置避免硬编码在多个节点中。
- 创建包、接口或构建配置前，先确认目标 ROS 2 发行版、构建类型以及当前目录结构。
- 运行或构建 ROS 2 工作空间前，确认并报告实际使用的 ROS 环境与 overlay；不得编造未验证的安装、构建或测试命令。
- 当前没有已验证的构建、运行或测试命令。后续只能在相应的包、构建配置或测试文件实际存在后，将其真实命令补充到本文件。
- 排查 ROS 2 问题时，优先检查环境、节点状态、Topic、TF、时间戳、QoS 和参数。

## Python 环境

- 本项目使用项目中的 `.venv`。
- 执行 Python、pip 和测试命令前，必须确认使用 `.venv` 中的解释器。
- 优先使用：
  - `.venv/bin/python`
  - `.venv/bin/python -m pip`
  - `.venv/bin/python -m pytest`
- 不要使用 `sudo pip` 或 `pip install --user`。
